# React и модуль сделок: расширенный учебный разбор

Ниже — детальный материал для студентов. Мы по шагам разберем, как React изменил фронтенд и почему он упрощает жизнь на примере текущего модуля сделок. В конце — лабораторная работа с пошаговой инструкцией миграции на React, где у каждого шага есть цель и смысл.

---

## 1. Что именно изменил React во фронтенде — подробно и на примерах

### 1.1 Состояние как источник истины (UI = f(state))
**Идея:** интерфейс — это функция от состояния. Если меняются данные, UI пересчитывается автоматически.

**Vanilla JS (фрагмент из проекта):**
- Данные приходят из API → мы вручную рисуем HTML (`renderDeals`).
- Потом, чтобы изменить статус, нужно «находить» нужный DOM-элемент и менять его вручную или перерендерить всё.

**React-подход:**
- Есть `state` (например, `deals`).
- Компонент рендерит карточки из `deals`.
- Меняем `state` → React сам обновит DOM.

**Мини-пример:**
```jsx
const [deals, setDeals] = useState([]);
return deals.map(d => <DealCard key={d.id} deal={d} />);
```

**Почему это важно:**
- Нет «двух правд» (данные в JS и HTML).
- Не нужно руками искать и править DOM.

---

### 1.2 Декларативный UI вместо императивного DOM
**Идея:** описываем, *как должно выглядеть*, а не *как это рисовать*.

**Vanilla JS:**
- `createElement`, `innerHTML`, `appendChild`, `addEventListener`.
- Ручная логика вставки/удаления узлов.

**React:**
- JSX: `return (<div>{deal.title}</div>)`
- React сам создает/удаляет DOM-узлы.

**Мини-пример:**
```jsx
return (
  <div className="deal-card">
    <div className="deal-title">{deal.title}</div>
  </div>
);
```

**Почему это важно:**
- Код легче читать и поддерживать.
- UI «следует» данным автоматически.

---

### 1.3 Компонентная архитектура
**Идея:** интерфейс = набор независимых компонентов.

**Vanilla JS:**
- В `deals.js` всё смешано: загрузка данных, рендер карточек, drag&drop, модалка.

**React:**
- Компоненты: `DealsPage`, `KanbanColumn`, `DealCard`, `DealModal`.
- Каждый компонент отвечает за свою часть.

**Мини-пример:**
```jsx
<DealsPage>
  <KanbanColumn status="new" deals={...} />
  <KanbanColumn status="won" deals={...} />
</DealsPage>
```

**Почему это важно:**
- Легко переиспользовать карточку сделки в другом месте.
- Код становится модульным и масштабируемым.

---

### 1.4 Однонаправленный поток данных
**Идея:** данные идут сверху вниз, через `props`.

**Vanilla JS:**
- Глобальные переменные (`clients`).
- Скрипты зависят от порядка загрузки.

**React:**
- `DealsPage` получает `deals` в `state`.
- Передает их в `KanbanColumn` через `props`.

**Мини-пример:**
```jsx
<KanbanColumn deals={deals} onDropDeal={updateStatus} />
```

**Почему это важно:**
- Нет скрытых зависимостей.
- Легче тестировать и отлаживать.

---

### 1.5 Минимальные изменения DOM (diffing)
**Идея:** React сравнивает предыдущее и новое состояние UI и изменяет только то, что нужно.

**Vanilla JS:**
- `renderDeals` сначала чистит все колонки и рисует заново.

**React:**
- Меняем статус одной сделки → React перестроит только ее карточку.

**Почему это важно:**
- Быстрее на больших списках.
- Нет «мерцания».

---

## 2. Слабые места Vanilla JS в модуле сделок — подробно и с примерами

Ниже — прямое сопоставление с вашим кодом `frontend/js/deals.js`.

### 2.1 Полная перерисовка колонок
**Где в коде:**
- `renderDeals()` очищает DOM и вставляет карточки заново.

**Проблема:**
- Перерисовываются даже неизменные карточки.
- При 200+ сделках это «убивает» производительность.

**React-решение:**
- Обновляется только изменившаяся карточка.

---

### 2.2 Состояние разнесено и неявно
**Где в коде:**
- Нет общего массива `deals` в памяти.
- Данные «живут» только в DOM.

**Проблема:**
- Любое действие требует поиска DOM-элемента.
- Легко получить несоответствие между данными и отображением.

**React-решение:**
- `const [deals, setDeals] = useState([])`.

---

### 2.3 Глобальные зависимости
**Где в коде:**
- `clients` загружается в `clients.js`.
- `deals.js` использует глобальный `clients`.

**Проблема:**
- Ошибка, если модалка открылась раньше загрузки клиентов.
- Скрипты должны быть подключены в строгом порядке.

**React-решение:**
- В `DealsPage` можно загрузить `clients` через отдельный хук и передать в модалку.

---

### 2.4 Ручная работа с событиями
**Где в коде:**
- `addEventListener` для drag&drop и формы.

**Проблема:**
- Вся логика в одном файле.
- Сложно повторно использовать карточку/колонку.

**React-решение:**
- `onDragStart`, `onDrop` как пропсы компонентов.

---

### 2.5 Лишние запросы и отсутствие оптимизма
**Где в коде:**
- `updateDealStatus` вызывает `loadDeals()`, что перезагружает список.

**Проблема:**
- Лишний сетевой трафик.
- UI обновляется только после ответа сервера.

**React-решение:**
- Оптимистичное обновление: сразу меняем `state`, а сервер подтверждает позже.

---

## 3. Как React решает эти проблемы — связка «проблема → решение»

| Проблема | Vanilla JS | React |
|---------|------------|-------|
| Перерисовка | Полный пересчет DOM | Обновление минимальных узлов |
| Состояние | Нет единого источника | `state` как источник истины |
| Связность | Глобальные переменные | `props` и локальные состояния |
| События | Ручные `addEventListener` | Декларативные `onClick`, `onDrop` |
| UX | Задержки, мерцания | Мгновенный отклик |

## 5. Примеры React-реализации (с пояснениями)

### 5.1 Хук для загрузки сделок
```jsx
import { useEffect, useState } from 'react';
import { apiGet, apiPut, apiPost } from '../api/apiClient';

export function useDeals() {
  const [deals, setDeals] = useState([]);
  const [loading, setLoading] = useState(true);

  async function loadDeals() {
    setLoading(true);
    try {
      const data = await apiGet('/api/deals/');
      setDeals(data);
    } finally {
      setLoading(false);
    }
  }

  async function updateStatus(id, status) {
    setDeals(prev => prev.map(d => d.id === id ? { ...d, status } : d));
    await apiPut(`/api/deals/${id}`, { status });
  }

  async function createDeal(payload) {
    await apiPost('/api/deals/', payload);
    await loadDeals();
  }

  useEffect(() => { loadDeals(); }, []);

  return { deals, loading, updateStatus, createDeal, reload: loadDeals };
}
```

**Пояснение:**
- `useState` хранит массив сделок.
- `updateStatus` меняет UI сразу.
- `useEffect` запускает загрузку при первом рендере.

---

### 5.2 Колонка
```jsx
export default function KanbanColumn({ status, deals, onDropDeal }) {
  function handleDrop(e) {
    e.preventDefault();
    const dealId = Number(e.dataTransfer.getData('text/plain'));
    if (!dealId) return;
    onDropDeal(dealId, status);
  }

  return (
    <div className="kanban-column" data-status={status}>
      <div className="column-header">
        <h3>{status}</h3>
        <span className="deal-count">{deals.length}</span>
      </div>
      <div
        className="column-content"
        onDragOver={e => e.preventDefault()}
        onDrop={handleDrop}
      >
        {deals.map(deal => <DealCard key={deal.id} deal={deal} />)}
      </div>
    </div>
  );
}
```

**Пояснение:**
- Колонка получает только свои сделки.
- Drag&drop логика изолирована внутри компонента.
